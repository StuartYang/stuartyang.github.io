《为什么废弃永久代》http://openjdk.java.net/jeps/122

本文主要讲述JVM的 Run-Time Data Areas，俗称JVM内存结构。

  • 程序计数器：每个 Java 虚拟机线程都有自己的 pc（程序计数器）寄存器。在任何时候，每个 Java 虚拟机线程都在执行单个方法的代码。
    • 如果该方法不是 native，则该pc寄存器包含当前正在执行的 Java 虚拟机指令的地址。
    • 如果线程当前正在执行的方法是native，则 Java 虚拟机pc 寄存器的值是未定义的。Java 虚拟机的pcregister 足够宽，可以returnAddress在特定平台上保存一个或一个本地指针。
  • 栈：伴随着线程的诞生，每个线程都有一个私有的栈被创建。栈上存储栈帧。栈上保存局部变量和部分结果，并在方法调用和返回中发挥作用。栈的内存不需要是连续的。栈具有固定大小或根据计算需要动态扩展和收缩
    • 固定：如果线程请求的栈深度大于虚拟机所设定的深度，则Java虚拟机将引发StackOverflowError。
    • 动态：如果虚拟机在扩展栈时无法申请到足够的内存空间，则抛出OutOfMemoryError异常。
  • 堆：堆一个在所有JVM中线程之间共享的空间。堆是运行时数据区，从中分配所有类实例和数组的内存。
    • 堆是在虚拟机启动时创建的。
    • 自动存储管理系统（称为垃圾收集器）回收
    • 堆的内存不需要是连续的。
    • 用户通过控制最大和最小堆大小调控堆的大小。
    • 若计算所需要的堆多于自动存储管理系统所能提供的堆，Java 虚拟机将抛出一个 OutOfMemoryError。
  • 方法区：线程之间共享的数据空间。它存储每个类的结构（类信息）。例如：运行时常量池、字段和方法数据，以及方法和构造函数的代码，包括在类和实例初始化和接口初始化中使用的特殊方法。
    • 方法区是在虚拟机启动时创建的。
    • 逻辑上是堆的一部分。
    • 实现上可能会选择不进行垃圾收集或压缩它。
    • 方法区的内存不需要是连续的。
    • 方法区可以是固定大小，也可以根据计算的需要进行扩展。
    • 用户可以通过对最大和最小方法区大小的控制方法区的大小。
    • 如果方法区中的内存不能用于满足分配请求，Java 虚拟机将抛出一个OutOfMemoryError。
  • 运行时常量池：
    • 包含了每个类或每个接口中常量表中运行时的体现。运行时常量池的功能类似于传统编程语言的符号表。
      • 类或接口的运行时常量池是在Java 虚拟机创建类或接口时构造的。
      • 在创建类或接口时，构建运行时常量池需要的内存多于 Java 虚拟机的方法区中可用的内存，则 Java 虚拟机将抛出一个OutOfMemoryError.
  • 本地方法栈：
    • 本机方法栈也由 JVM的解释器执行。
      • 创建每个线程时，为每个线程分配本机方法堆栈。
      • 本地方法栈可以是固定大小或根据计算需要动态扩展和收缩
      • 用户控制本地方法堆栈的初始大小，最大和最小。
      • 如果线程中的计算需要比允许的更大的本机方法堆栈，则 Java 虚拟机将抛出StackOverflowError.
      • 如果可以动态扩展本机方法堆栈并尝试扩展本机方法堆栈但可以提供足够的内存，或者如果可以提供足够的内存来为新线程创建初始本机方法堆栈，则 Java 虚拟机将抛出OutOfMemoryError.
  • 栈帧：
    • 用于存储数据和部分结果，运行时常量池的引用，对于方法返回值，并调度异常。
      • 调用方法时都会创建一个新栈帧。其方法调用完成时被销毁。
      • 帧是从创建帧的线程的 Java 虚拟机栈。
      • 每个帧都有自己的局部变量数组、操作数栈，对当前方法类的运行时常量池的引用。
        • 局部变量数组的大小和操作数栈的最大深度在编译时确定
        • 局部变量（数组）：存储基本数据类型，引用类型和返回地址。
        • 操作数栈：将当前帧的操作数堆栈简称为操作数堆栈。
        • 创建包含它的帧时，操作数堆栈为空。
        • 运行时常量池的引用
        • 调用者的程序计数器适当增加以跳过方法调用指令。然后在调用方法的帧中继续正常执行，并将返回值（如果有）推送到该帧的操作数堆栈上。
        • 异常发生，永远不会向其调用者返回值。
      • 由线程创建的帧是该线程的本地帧，不能被任何其他线程引用。

-XX:MaxMetaspaceSize，最大空间，默认是没有限制的。如果没有指定这个参数，元空间会在运行时根据需要动态调整。

  1. 字符串存在永久代中，容易出现性能问题和内存溢出。

　　2、类及方法的信息等比较难确定其大小，因此对于永久代的大小指定比较困难，太小容易出现永久代溢出，太大则容易导致老年代溢出。默认情况下，元空间的大小仅受本地内存限制

　　3、永久代会为 GC 带来不必要的复杂度，并且回收效率偏低。

　　4、JRockit 没有永久代，Oracle能会将HotSpot 与 JRockit 合二为一。

对于僵死的类及类加载器的垃圾回收将在元数据使用达到“MaxMetaspaceSize”参数的设定值时进行。

元空间是方法区在HotSpot JVM 中的实现，方法区主要用于存储类的信息、常量池、方法数据、方法代码等。方法区逻辑上属于堆的一部分，但是为了与堆进行区分，通常又叫“非堆”。

元空间的本质和永久代类似，都是对JVM规范中方法区的实现。不过元空间与永久代之间最大的区别在于：元空间并不在虚拟机中，而是使用本地内存。理论上取决于32位/64位系统可虚拟的内存大小，可见也不是无限制的，需要配置参数。

初始化的Metaspace大小，控制元空间发生Metaspace GC的阈值。Metaspace GC后，动态增加或降低MetaspaceSize。

对于僵死的类及类加载器的垃圾回收将在元数据使用达到MaxMetaspaceSize参数的设定值时进行, 进行的是full gc。

适时地监控和调整元空间对于减小垃圾回收频率和减少延时是很有必要的。持续的元空间垃圾回收说明，可能存在类、类加载器导致的内存泄漏或是大小设置不合适。

  1. 字符串常量由永久代转移到堆中，因为字符串存在永久代中，容易出现性能问题和内存溢出。
  2. 持久代已不存在，PermSize 和 MaxPermSize 参数已移除。
